# Unveiling Serialization Bottlenecks: JSON vs. Protocol Buffers

This repository contains the source code, micro-benchmarking suite, and open data logs for the research paper: **"Unveiling Serialization Bottlenecks: An Empirical Evaluation of Garbage Collection Bloat, Tail Latency, and Payload Efficiency in JSON versus Protocol Buffers."**

## Overview
In modern distributed systems, data serialization efficiency dictates system performance. This project utilizes the **Java Microbenchmark Harness (JMH)** to isolate and empirically evaluate the algorithmic execution, spatial footprint, and Garbage Collection (GC) degradation of **JSON (Jackson)** versus **Protocol Buffers (Protobuf)**.

To provide a comprehensive analysis simulating real-world databases, the benchmark tests three distinct structural complexities using 100,000 synthetically generated records:
1. **Level 1 (Flat):** Simple key-value pairs containing only primitive data types.
2. **Level 2 (Shallow):** Objects containing a single layer of arrays.
3. **Level 3 (Deeply Nested):** Complex, multi-layered polymorphic structures.

## Prerequisites
To replicate this benchmark on your local machine, ensure you have the following installed:
* Java Development Kit (JDK) 25 or higher
* Apache Maven (3.6.3 or higher)
* Protocol Buffers Compiler (`protoc`)

## How the Benchmark Works
The benchmarking framework is divided into two main execution phases to prevent JIT compiler optimization biases:

1. **JMH Execution (Speed & Garbage Collection):** 
   Executes 3 warmup iterations to stabilize the JVM, followed by 5 measurement iterations. It uses JMH's compiler blackholes to consume serialized outputs and attaches the `-prof gc` profiler to monitor memory allocation rates and Garbage Collection (GC) bloat.
2. **Payload Size Execution (`SizeMeasurer`):** 
   A secondary script that performs a single serialization pass to calculate the absolute network payload size (in bytes) of the encoded data.

## How to Run the Benchmark

**Step 1: Clone the repository and build the project**
```bash
git clone [https://github.com/renerdre/serialization-bottlenecks-jmh](https://github.com/renerdre/serialization-bottlenecks-jmh)
cd protobuf-vs-json-microbenchmark
mvn clean install
```

**Step 2: Run the JMH Latency & GC Profiler**
This command executes the benchmark and profiles the Garbage Collection memory bloat.
```bash
java -jar target/benchmarks.jar -prof gc
```

**Step 3: Run the Payload Size Measurer**
This command measures the exact byte footprint generated by each serialization protocol.
```bash
mvn exec:java "-Dexec.mainClass=com.serialization.benchmark.SizeMeasurer"
```

## Key Results Summary

Our empirical micro-benchmarking revealed the following critical insights:

* **Payload Size Efficiency:** Protobuf consistently minimizes bandwidth. In Level 3 (Deeply Nested) schemas, Protobuf reduced the payload size by **62%** (from 1,963 bytes down to 740 bytes) compared to JSON.
* **Execution Speed:** Protobuf universally outperformed JSON. In complex Level 3 data, Protobuf was approximately **3.1x faster** (1.076 µs vs JSON's 3.399 µs).
* **The Hidden GC Bottleneck:** JSON exhibits severe memory bloat in complex structures. For Level 3 data, JSON discarded **4,627 bytes** of memory per execution compared to Protobuf's 804 bytes. This caused JSON's P99.9 worst-case tail latency to spike to **34.176 µs** due to GC-induced lag.
* **The Flat Data Trade-off:** While Protobuf mathematically outperforms JSON in Level 1 (Flat) schemas, the absolute temporal gain is a negligible **0.228 microseconds**. For simple architectures, this microscopic advantage does not justify the loss of human readability and strict schema maintenance, proving JSON remains practically superior for flat data.
